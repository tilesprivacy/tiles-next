# Manual

## Command Line Options

Commands are grouped by subcommand family: the base `tiles` command, models, identity, data layout, updates, ATProto linking for sharing, then peer-to-peer linking and sync.

### Main command (`tiles`)

Launches onboarding (first run) and the interactive REPL.

```bash
tiles   # Onboarding (if needed) and REPL
```

### Run models (`tiles run`)

Start a chat session with a model; optional Modelfile path.

```bash
tiles run [MODELFILE_PATH]   # Default model if path omitted
```

```bash
tiles run
tiles run ./path/to/Modelfile
```

Create a new Modelfile by adding a plain text file named `Modelfile`:

```text
FROM mlx-community/Qwen3.5-4B-MLX-4bit
SYSTEM You are Mario from Super Mario Bros, acting as an assistant.
```

Use any `mlx-community/` model hosted on Hugging Face in the `FROM` field. Other model hosts and namespaces are not supported there right now. Learn more in the [Tilekit Modelfile reference](/book/tilekit).

### Account (`tiles account`)

Root identity and display nickname.

```bash
tiles account create [<nickname>]  # Create root identity (nickname optional)
tiles account set-nickname         # Set nickname
tiles account                      # Show account details
```

```bash
tiles account create alice
tiles account set-nickname
tiles account
```

### Data folder (`tiles data`)

Where Tiles stores local data files.

```bash
tiles data set-path <path>   # Set data folder
```

```bash
tiles data set-path ~/.tiles/data
```

### App updates (`tiles update`)

```bash
tiles update   # Check for and install app updates
```

### Peer-to-peer link and sync (`tiles link / sync`)

Tiles can sync chats across multiple **linked** devices peer-to-peer, with or without the internet. Linking is a one-time step so both sides consent before any sync traffic is accepted; you do not need to re-link each time you sync.

Online and offline linking use the same commands. Iroh handles transport in both cases. On the same local network without internet, devices discover each other over mDNS.

#### Link devices

On the first device:

```bash
$ tiles link did:key:z6Mknxy...YWSmTcZA
```

The command prints a token in the terminal. Share it out of band with the peer you want to link.

On the other device:

```bash
tiles link add <token>
```

The token format depends on how the devices connect:

- **Online**: a **UCAN** token (for example, `ucan:eyJ...YzQ`).
- **Offline** (same network, mDNS discovery via Iroh): an **eight-character alphanumeric code**.

After the peer runs **`tiles link add`**, approve the link request on the first device. The two devices are then linked.

**`tiles link <DID>`**: Starts linking toward the peer with that DID and prints the token for the other device.

**`tiles link add <token>`**: Completes linking with the token from the first device's output.

**`tiles link list-peers`**: Lists linked peers by **DID** (decentralized identifier) and **nickname**.

```bash
tiles link list-peers
```

#### Sync chats

Once linked, run:

```bash
$ tiles sync did:key:z6Mknxy...YWSmTcZA
```

**`tiles sync <DID>`**: Starts a sync run toward the linked peer with that DID.

**Online**: run **`tiles sync <DID>`** on either device when you want to sync.

**Offline** (same network): sync is synchronous and uses a listener and initiator. On the device that will receive the sync, start the listener first:

```bash
tiles sync        # Listener: wait for incoming sync requests
```

On the other device, run the initiator toward the listener's DID:

```bash
$ tiles sync did:key:z6Mknxy...YWSmTcZA
```

When a sync run finishes, you should get a notification on the devices involved.

#### Unlink a peer (`tiles link revoke`)

**`tiles link revoke <DID>`**: Unlinks a peer. Future sync requests from that peer are ignored.

```bash
tiles link revoke <DID>   # Unlink a peer; ignore their future sync requests
```

### ATProto (`tiles at`)

ATProto is currently used to share chat sessions.

- `tiles at login <handle>`: log in with your ATProto handle (for example, `alice.bsky.social`) via browser OAuth.
- `tiles at logout`: log out of the current ATProto account.

```bash
tiles at login alice.bsky.social
tiles at logout
```

Authentication currently uses a localhost callback (`127.0.0.1:8988`), and session state is stored locally in the Tiles database.

## Slash commands

Slash commands give you fast, keyboard-first control over Tiles in interactive chat. Type `/?` or `/help` in the prompt to open available commands and quickly run actions like checking session state, resuming prior sessions, and sharing chats without leaving the terminal. Use `/bye` to exit the chat session.

### Session Commands

#### `/status`

Show the current session state.

#### `/sessions`

List available sessions in local storage.

#### `/resume <sessionId>`

Resume a specific session by ID.

### Sharing Commands

These commands require an active ATProto login. See [#ATProto (`tiles at`)](#atproto-tiles-at).

#### `/share` (via ATProto)

Create a shareable link for the current session.

#### `/share <sessionId>` (via ATProto)

Create a shareable link for a specific session.

Example shared session link: [tiles.run/share/YXQ6Ly9k...xeTI3](/share/YXQ6Ly9kaWQ6cGxjOm1iazZ3Z214aWF0b3R6eTViM3E1N25hdy9ydW4udGlsZXMuc2Vzc2lvbi8zbWtuMm9veG5xeTI3).

## Uninstall Tiles (macOS)

By default, uninstalling Tiles removes only binaries and shared payloads under `/usr/local`. User data files under `~/.local/share/tiles` (or a custom `[data] path` in `config.toml`) are not deleted, so chats, memory, databases, and logs remain for a later reinstall. Wipe that data only when you want a clean slate.

Read the full step-by-step procedure in the [Tiles uninstaller skill](https://tiles.run/tiles-uninstaller-skill/SKILL.md), including optional receipt and keychain cleanup. With coding agents (for example Claude Code or Codex), share that URL in your prompt and ask the agent to follow it, adapting paths that differ on your machine.

We are actively working on an official Tiles uninstaller.
