## Tiles

Commands in this section are for regular users. Most day-to-day behavior is also available as slash commands inside the chat REPL.

Commands are grouped by subcommand family: the base `tiles` command, identity, data layout, models, updates, then peer-to-peer linking and sync.

### Account (`tiles account`)

Root identity and display nickname.

```bash
tiles account create [<nickname>]  # Create root identity (nickname optional)
tiles account set-nickname         # Set nickname
tiles account                      # Show account details
```

```bash
tiles account create alice
tiles account set-nickname
tiles account
```

### Data directory (`tiles data`)

Where Tiles stores local data.

```bash
tiles data set-path <path>   # Set data directory
```

```bash
tiles data set-path ~/.tiles/data
```

### Run models (`tiles run`)

Start a chat session with a model; optional Modelfile path and experimental memory flag.

```bash
tiles run [MODELFILE_PATH]   # Default model if path omitted
tiles run -m                 # Experimental memory model
```

```bash
tiles run
tiles run ./path/to/Modelfile
tiles run -m
```

### App updates (`tiles update`)

```bash
tiles update   # Check for and install app updates
```

### Peer-to-peer: linking (`tiles link`)

Tiles can sync chats across multiple **linked** devices peer-to-peer, with or without the internet. Linking is a one-time step so both sides consent before any sync traffic is accepted; you do not need to re-link each time you sync.

```bash
tiles link enable      # Become link listener; share the ticket out of band
tiles link list-peers  # Show linked peers (DID and nickname)
tiles link disable     # Unlink a peer; ignore their future sync requests
```

**`tiles link enable`**: Puts this device in **link listener** mode. It generates a **link ticket** and waits for a link request from another device. Share the ticket **out of band** with the peer you want to link.

- If the device is **online**, the ticket is a **base64** string (easy to copy).
- If **offline**, the ticket is an **eight-character alphanumeric code** you can type on another device on the same network.

When the peer runs **`tiles link enable`** and supplies your ticket, this device is notified; after you **approve**, the two devices are linked.

**`tiles link list-peers`**: Lists linked peers by **DID** (decentralized identifier) and **nickname**.

**`tiles link disable`**: Unlinks a peer. Future sync requests from that peer are ignored.

### Peer-to-peer: chat sync (`tiles sync`)

After devices are linked, use **`tiles sync`** to replicate chats. Listener and initiator roles mirror linking: one side waits, the other connects using the listener’s DID.

```bash
tiles sync        # Listener: wait for incoming sync requests
tiles sync <DID>  # Initiator: sync toward the peer with this DID (the listener)

# Example
tiles sync did:key:z6Mknxy...YWSmTcZA
```

**`tiles sync`** (listener): Starts this device in **sync listener** mode: it waits for incoming sync requests.

On the **other** device, run **`tiles sync`** with the listener’s **DID** so the initiator targets the correct peer. If your CLI uses different syntax for the peer argument, check `tiles sync --help`.

When a sync run finishes, you should get a notification on the devices involved.

## Uninstall Tiles (macOS)

This section is **macOS only**. Tiles does not ship a built-in uninstaller. You stop the app, remove the files the installer placed under `/usr/local`, remove your data under your home folder (and any custom paths), then optionally clear Keychain entries and Installer receipts.

### How the macOS `.pkg` installs Tiles

Understanding the installer makes uninstall predictable.

#### What Apple’s Installer does

The release ships one or more **component packages**. For the full downloadable product, those are wrapped into the installer you download from the site.

| Package identifier | Role |
| --- | --- |
| `com.tilesprivacy.tiles` | Main app: CLI binary, Python server stack under `server/`, `modelfiles/`, and related files |
| `com.tilesprivacy.tiles_models` | **Offline (“Tiles Offline”) bundle only:** extra payload (bundled models) under the same `/usr/local/share/tiles` tree |

- **Network-style installer:** installs **only** `com.tilesprivacy.tiles`. Models are fetched later by Tiles.
- **Offline installer:** can install **both** the app package and `com.tilesprivacy.tiles_models` in one run.

The binary is installed to `/usr/local/bin/tiles`, and shared files to `/usr/local/share/tiles` (server, modelfiles, and any bundled models).

#### Scripts that run during install

- **Preinstall:** removes the previous `/usr/local/share/tiles/server` tree so upgrades do not leave duplicate Python stacks.
- **Postinstall:** unpacks and runs the embedded Python stack’s post-install steps under `/usr/local/share/tiles/server`.

Installer.app records **receipts** (package IDs) in the system database. Receipts are **not** removed when you delete files by hand, so after a manual uninstall you may want to forget them (see step 3 below).

### 1. Stop Tiles

```bash
tiles server stop
tiles daemon stop
```

If `tiles` is already gone, skip this or quit any leftover Tiles-related processes in **Activity Monitor**.

### 2. Remove files installed under `/usr/local`

This matches what the `.pkg` lays down (and the same paths used by the shell install script if you installed that way):

```bash
sudo rm -f /usr/local/bin/tiles
sudo rm -rf /usr/local/share/tiles
```

**Optional:** see exactly what the receipt tracked (useful if you changed layouts or are verifying):

```bash
pkgutil --pkgs | grep tilesprivacy
pkgutil --files com.tilesprivacy.tiles
# If you used the offline installer with bundled models:
pkgutil --files com.tilesprivacy.tiles_models
```

`pkgutil --files` lists paths as installed; removing `/usr/local/share/tiles` and the binary still covers a normal install.

### 3. Optional: forget Installer receipts

After the files above are gone, you can drop the records so macOS no longer thinks those packages are installed:

```bash
sudo pkgutil --forget com.tilesprivacy.tiles
sudo pkgutil --forget com.tilesprivacy.tiles_models
```

The second line applies only if `com.tilesprivacy.tiles_models` appears in `pkgutil --pkgs`. `--forget` does **not** delete files; it only updates the receipt database (use it after you have already removed the payload, or you will have orphaned files).

### 4. Remove your data (config, logs, databases, memory)

Tiles uses XDG-style paths. On a typical Mac shell, `XDG_CONFIG_HOME` and `XDG_DATA_HOME` are unset, so defaults are:

| Purpose | Default on Mac |
| --- | --- |
| Config (`config.toml`, `server.pid`, …) | `~/.config/tiles` |
| Data (logs, default `data/` including memory, downloaded models under your data dir, …) | `~/.local/share/tiles` |

If you set `XDG_CONFIG_HOME` or `XDG_DATA_HOME` in `~/.zprofile` / `~/.zshrc`, remove the `tiles` folder under those locations instead.

```bash
rm -rf ~/.config/tiles
rm -rf ~/.local/share/tiles
```

Or, in one shot respecting XDG:

```bash
rm -rf "${XDG_CONFIG_HOME:-$HOME/.config}/tiles"
rm -rf "${XDG_DATA_HOME:-$HOME/.local/share}/tiles"
```

#### Custom `[data] path`

If `config.toml` sets a non-empty `[data] path`, databases and related data live there. Check before deleting:

```bash
cat ~/.config/tiles/config.toml
```

(If you already removed `~/.config/tiles`, recall your custom path from backup or notes.)

### 5. Optional: Hugging Face cache

Tiles may still resolve some models under `~/.cache/huggingface/hub`. Deleting that directory affects **every** app on this Mac that uses the Hugging Face cache, not only Tiles.

### 6. Optional: Keychain

Secrets (for example the database encryption key and identity material) are stored via the system keychain under the service name `tiles`. Open **Keychain Access**, search for **tiles**, and delete matching entries if you want credentials removed.

### 7. Optional: `PATH` in `~/.zshrc`

Only if you explicitly added a line for Tiles; most Macs already pick up `/usr/local/bin`.

### 8. Optional: development builds on this Mac

Debug builds from a source tree use `.tiles_dev/tiles` next to the project instead of `~/.config` / `~/.local/share`. Remove that folder if you used a local debug binary.

> **Warning:** `rm -rf` **permanently deletes** data: chats, encrypted databases, logs, local memory (PKM), and anything else under those paths. Confirm backups before running destructive commands.
